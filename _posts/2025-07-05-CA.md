## Introduction

In today's threat landscape, sophisticated adversaries often leverage social engineering and weaponized documents as their initial attack vector. Understanding how these attacks unfold is crucial for organizations to strengthen their security posture against real-world attack scenarios. By analyzing the tactics, techniques, and procedures (TTPs) of actual threat actors, security professionals can gain invaluable insights into defensive gaps and help build stronger security resilience.

One of the most prevalent initial access methods observed in the wild is phishing campaigns utilizing malicious Microsoft Office documents. These attacks prey on human psychology and the ubiquitous nature of office productivity suites in corporate environments. A seemingly innocent resume or business proposal can quickly become the gateway for sophisticated malware deployment.

In this demonstration, we'll walk through how a typical adversary attack chain weaponizes everyday documents to establish persistent command and control channels. Our scenario showcases how attackers would leverage:

1. **Initial Payload**: A weaponized Word document (`Resume.docm`) containing an embedded VBA macro
2. **Loader Mechanism**: A PowerShell-based loader (`loader.ps1`) designed to retrieve and execute additional payloads
3. **Encoded Shellcode**: XOR-encoded shellcode (`demon_encoded.x64.bin`) hosted on the attacker's infrastructure
4. **Command & Control**: A Havoc C2 framework providing post-exploitation capabilities

This attack chain demonstrates several key techniques commonly observed in real-world intrusions: macro-enabled document exploitation, fileless execution through PowerShell, payload encoding for evasion, and in-memory shellcode execution to avoid disk-based detection mechanisms.

## Initial Payload: The Trojan Horse

At first glance, our weaponized document appears as an innocent resume the perfect social engineering lure. However, beneath this benign facade lies a sophisticated attack mechanism waiting to be unleashed. This is the digital equivalent of a Trojan horse, designed to exploit the most vulnerable component in any security infrastructure: human trust.

The document leverages Microsoft Word's macro functionality, a legitimate feature that becomes a powerful attack vector in the wrong hands. When a victim opens the document and enables macros often prompted by social engineering text claiming the document is "protected" or "encrypted" the embedded VBA code springs into action.

Use `Alt + F11` to access the VBA editor and examine the macro code hidden within any `.docm` file.

#### Macro Activation: The Point of No Return

Once the user clicks "Enable Content," the malicious macro executes silently in the background. This moment represents the initial compromise the attacker has successfully breached the perimeter and established their first foothold within the target environment.

**PowerShell Execution**  
The macro constructs a PowerShell command with specific evasion parameters:
- `-w hidden`: Runs PowerShell with a hidden window to avoid detection
- `-nop`: Disables PowerShell profiles that might contain security restrictions
- `-enc`: Executes the Base64-encoded command directly

**Silent Execution**  
Finally, `Shell cmd, vbHide` launches the PowerShell process invisibly in the background, making the attack completely transparent to the victim.

This elegant yet malicious code demonstrates how attackers can weaponize legitimate Office features to establish initial access with minimal detection risk.

### The Cascading Download Chain

Once the victim opens the Word document and activates the macro, a sophisticated multi-stage payload delivery unfolds:

**Stage 1: Initial Callback**  
The compromised system makes an HTTP GET request to retrieve `loader.ps1` from the attacker's web server. This PowerShell script serves as the secondary payload loader, designed to orchestrate the next phase of the attack.

**Stage 2: Shellcode Retrieval**  
The loader script subsequently requests `demon_encoded.x64.bin` an XOR encoded shellcode payload. This encoding technique serves multiple purposes:
- **Evasion**: Bypasses signature-based detection mechanisms
- **Obfuscation**: Hides the true nature of the payload during transit
- **Anti-Analysis**: Complicates reverse engineering efforts

**Stage 3: In-Memory Execution**  
Upon successful retrieval, the loader decodes the XOR-encrypted shellcode and executes it directly in memory, avoiding disk-based artifacts that could trigger security solutions.

### Establishing Command & Control

The decoded shellcode establishes a reverse connection back to the attacker's Havoc C2 framework, completing the initial compromise phase. This connection provides the adversary with:

- **Remote Command Execution**: Full shell access to the compromised system
- **Persistence Mechanisms**: Ability to maintain long-term access
- **Lateral Movement**: Capability to expand access within the network
- **Data Exfiltration**: Channels for stealing sensitive information
