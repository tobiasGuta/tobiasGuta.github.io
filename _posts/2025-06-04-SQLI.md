# SQL (Structured Query Language)
--------------------------------------

SQL is a powerful language used to interact with relational databases. It lets you fetch, update, insert, and remove data using clear commands called **statements**.

In this guide, we'll stick with **MySQL-style syntax**. It's important to know SQL is **not case-sensitive**, so commands like `SELECT`, `select`, and `SeLeCt` all behave the same.

# What is SQL Injection (SQLi)?
-----------------------------

**SQL Injection (SQLi)** is a vulnerability that occurs when an application takes **user input** and drops it directly into an **SQL query** without proper validation or sanitization. This opens the door for attackers to **manipulate the query** and extract, bypass, or destroy data.

It's like handing your database a loaded gun... and letting the user pull the trigger.

* * * * *

Real-World Example -- The Blog Post Trap
------------------------------------------

Let's imagine a blog site where each post has a unique ID. Posts may be *public* (ready to read) or *private* (still in draft).

A standard blog post URL might look like this:

Under the hood, the backend may run an SQL query like this:

  <span class="code-lang-tag">SQL</span>

-   `is_public=1` --- filters out private drafts.

-   `LIMIT 1` --- ensures only one row is returned.

If `post 4` is public, you see it. If it's private (`is_public=0`), you get a "not found" page.

⚙️ Why This Happens
-------------------

It all boils down to **trusting user input**. SQL engines don't care where the data comes from they just execute the final string. If the app doesn't escape or bind inputs safely, attackers write their own logic inside your query.


# In-Band SQL Injection: One Path In. One Path Out.
----------------------------------------------------

**In-Band SQLi** is the *loudest*, *simplest*, and *most direct* form of SQL Injection. It's called "In-Band" because:

> The **request** and **response** both travel the **same channel** usually the web browser.

You shoot the payload ➜ the app executes it ➜ the results spill out right on the page.

It's fast. It's obvious. And if it works, you're basically jacked into the DB like it's 1999.

# Boolean-Based Blind SQL Injection (SQLi)
----------------------------------------

Boolean-Based SQL Injection is a form of **blind injection** where the attacker doesn't see database output directly but instead infers it through differences in the application's behavior. The application's response is based on **true/false conditions**, allowing attackers to slowly extract data.

For example, a page might respond with "user found" when a condition is true, and "no user found" when false. That boolean switch is what we exploit.

* * * * *

# Understanding Second-Order SQL Injection: The Silent Threat

When we think about SQL injection (SQLi), the classic image is of attackers injecting malicious SQL commands directly into input fields causing immediate havoc like data leaks or corruption. But not all SQL injections flash their danger right away. Enter **Second-Order SQL Injection** a stealthier, more insidious cousin in the SQLi family.

What is Second-Order SQL Injection?
-----------------------------------

Second-Order SQL Injection (sometimes called stored SQLi) happens when malicious user input is saved in the database **without causing any immediate SQL errors or suspicious behavior**. The attack payload remains dormant harmless at first glance because the data isn't directly executed as SQL during initial insertion.

The real danger emerges **later**, when this stored data is retrieved and used in another SQL query **without proper sanitization** or escaping. This delayed injection point is what makes second-order SQLi tricky to detect and remediate.

In short:

-   The attacker injects **malicious SQL code into the database via one part of the app.**

-   The injected payload is stored as normal data.

-   Later, when the app uses this stored data in another SQL query (the "second order"), **the malicious payload executes**, enabling attacks like privilege escalation, data tampering, or data leakage.

* * * * *

Why Is This Attack Hard to Catch?
---------------------------------

Standard input validation or sanitization techniques often focus on preventing immediate SQLi input that breaks queries right away. But second-order SQLi bypasses this because the initial input looks harmless, causing no errors during insertion.

Detection systems monitoring input fields may see nothing wrong, and code reviewers might miss vulnerabilities in seemingly "safe" retrieval queries, assuming data in the database is clean.
