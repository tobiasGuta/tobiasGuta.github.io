# SQL (Structured Query Language)
SQL is a powerful language used to interact with relational databases. It lets you fetch, update, insert, and remove data using clear commands called **statements**.

In this guide, we'll stick with **MySQL-style syntax**. It's important to know SQL is **not case-sensitive**, so commands like `SELECT`, `select`, and `SeLeCt` all behave the same.

# What is SQL Injection (SQLi)?
-----------------------------

**SQL Injection (SQLi)** is a vulnerability that occurs when an application takes **user input** and drops it directly into an **SQL query** without proper validation or sanitization. This opens the door for attackers to **manipulate the query** and extract, bypass, or destroy data.

It's like handing your database a loaded gun... and letting the user pull the trigger.

* * * * *

Real-World Example -- The Blog Post Trap
------------------------------------------

Let's imagine a blog site where each post has a unique ID. Posts may be *public* (ready to read) or *private* (still in draft).

⚙️ Why This Happens
-------------------

It all boils down to **trusting user input**. SQL engines don't care where the data comes from they just execute the final string. If the app doesn't escape or bind inputs safely, attackers write their own logic inside your query.


# In-Band SQL Injection: One Path In. One Path Out.
----------------------------------------------------

**In-Band SQLi** is the *loudest*, *simplest*, and *most direct* form of SQL Injection. It's called "In-Band" because:

> The **request** and **response** both travel the **same channel** usually the web browser.

You shoot the payload ➜ the app executes it ➜ the results spill out right on the page.

It's fast. It's obvious. And if it works, you're basically jacked into the DB like it's 1999.

# Boolean-Based Blind SQL Injection (SQLi)
----------------------------------------

Boolean-Based SQL Injection is a form of **blind injection** where the attacker doesn't see database output directly but instead infers it through differences in the application's behavior. The application's response is based on **true/false conditions**, allowing attackers to slowly extract data.

For example, a page might respond with "user found" when a condition is true, and "no user found" when false. That boolean switch is what we exploit.

* * * * *

### Scenario

Imagine a web app that checks if a username exists:

-   ✅ `http://localhost:8080/user.php?id=admin` → Response: **user found**

-   ❌ `http://localhost:8080/user.php?id=admin123` → Response: **no user found**

# Understanding Second-Order SQL Injection: The Silent Threat

When we think about SQL injection (SQLi), the classic image is of attackers injecting malicious SQL commands directly into input fields causing immediate havoc like data leaks or corruption. But not all SQL injections flash their danger right away. Enter **Second-Order SQL Injection** a stealthier, more insidious cousin in the SQLi family.

What is Second-Order SQL Injection?
-----------------------------------

Second-Order SQL Injection (sometimes called stored SQLi) happens when malicious user input is saved in the database **without causing any immediate SQL errors or suspicious behavior**. The attack payload remains dormant harmless at first glance because the data isn't directly executed as SQL during initial insertion.

The real danger emerges **later**, when this stored data is retrieved and used in another SQL query **without proper sanitization** or escaping. This delayed injection point is what makes second-order SQLi tricky to detect and remediate.

In short:

-   The attacker injects **malicious SQL code into the database via one part of the app.**

-   The injected payload is stored as normal data.

-   Later, when the app uses this stored data in another SQL query (the "second order"), **the malicious payload executes**, enabling attacks like privilege escalation, data tampering, or data leakage.

* * * * *

Why Is This Attack Hard to Catch?
---------------------------------

Standard input validation or sanitization techniques often focus on preventing immediate SQLi input that breaks queries right away. But second-order SQLi bypasses this because the initial input looks harmless, causing no errors during insertion.

Detection systems monitoring input fields may see nothing wrong, and code reviewers might miss vulnerabilities in seemingly "safe" retrieval queries, assuming data in the database is clean.

# Bypassing WAFs and Filters in SQL Injection: A Tactical Breakdown

When dealing with Web Application Firewalls (WAFs) or filter-based defenses, **no single SQL injection technique is guaranteed to work across all environments**. Filters are often tailored to block known attack patterns but they're not perfect. Understanding their weaknesses and employing a flexible, adaptive mindset is key to bypassing them.

Below is a curated list of common filtering scenarios, along with practical evasion techniques and real-world examples:

| **Scenario** | **Description** | **Bypass Example** |
| --- | --- | --- |
| **Keywords like `SELECT` are blocked** | WAFs may block exact keyword matches. Bypass by changing case or inserting inline comments. | <code>SElEcT * FrOm users</code> or <code>SE/**/LECT * FROM/**/users</code> |
| **Spaces are filtered** | Spaces are removed or blocked. Use alternative whitespace (e.g., `%0A`, `%09`) or inline comments. | <code>SELECT%0A*%0AFROM%0Ausers</code> or <code>SELECT/**/username/**/FROM/**/users</code> |
| **Logical operators like `AND`, `OR` are banned** | Filters block logical operators. Use alternatives like `&&`, `||`, or encoded logic. | <code>username = 'admin' && password = 'pass'</code> or <code>username = 'admin'%0A</code> |
| **Keywords like `UNION`, `SELECT` are blocked** | Filters block keywords. Encode with `CHAR()`, Unicode, or hex to evade. | <code>SELECT * FROM users WHERE username = CHAR(0x61,0x64,0x6D,0x69,0x6E)</code> |
| **Deep keyword filtering (`OR`, `AND`, `SELECT`, `UNION`)** | Multiple keywords are filtered. Combine obfuscation: string functions, comments, concatenation. | <code>SELECT * FROM users WHERE username = CONCAT('a','d','m','i','n')</code> or <code>SElEcT/**/username/**/FROM/**/users</code> |

* * * * *

### Adapting to the Unknown: Patience and Precision

In real-world engagements, **you won't always have clear feedback** on what's being filtered or blocked. WAFs may silently drop requests, sanitize payloads, or serve generic error pages making your job harder. That's why **SQLi is often trial-and-error based**, especially under heavy defenses.

Each target environment is unique:

-   Some filter payloads at the frontend.

-   Others rely on backend input sanitization.

-   Many use both layered defenses.

To break through:

-   **Start small** test with basic payloads.

-   **Probe with variations** change syntax, encoding, and structure.

-   **Log everything** track responses, delays, and subtle changes.

It's a chess match, not a sprint. **WAFs can be defeated but only if you stay unpredictable.**

# Out-of-Band SQL Injection (OOB SQLi)

* * * * *

**What is Out-of-Band SQL Injection?**

Out-of-Band SQL Injection (OOB SQLi) is a next-level attack vector used when traditional SQL injection methods (like error-based or boolean-based) are blocked, sanitized, or heavily monitored. Instead of relying on the same channel for both the attack and the response, OOB SQLi splits the communication: you inject through one path, and extract data via another.

> Think: "I leave a hidden note in your house, but I read your reply off a billboard across town."

OOB SQLi turns external communication channels like DNS, HTTP, or SMB into covert exfiltration tunnels, letting attackers bypass firewalls, WAFs, and detection systems with surgical precision.

* * * * *

**Attack Flow: How OOB SQLi Operates**

1.  **Injection Point Identified:** Attacker finds a vulnerable SQL input.

2.  **Payload Delivered:** Payload is crafted to force the database to perform an outbound action.

3.  **External Callback:** Database calls an external system (DNS, HTTP, SMB) controlled by the attacker.

4.  **Exfiltration Complete:** The attacker sniffs the outbound request and extracts the data completely out-of-band.

This makes OOB SQLi perfect for stealthy ops in locked-down, segmented, or firewalled environments.

* * * * *

**Why Use OOB SQL Injection?**

Sometimes, you're up against:

-   **Read-only or sanitized output** (no direct response to your injection).

-   **Tight WAF/IDS filtering**, blocking classic error or time-based SQLi.

-   **Air-gapped** or segmented networks that block outbound HTTP but allow DNS or SMB.

-   **Stored procedures or encoding** that neuter direct payload feedback.

OOB is the ghost-in-the-wires solution it exploits internal-to-external communication channels, many of which are overlooked by defenders.

-   **X-Forwarded-For:** Client IP chain, often trusted blindly

-   **Referer:** Previous page URL, attacker-controlled

-   **Cookie:** Session data carriers

-   **X-Api-Key / Authorization:** Sometimes mishandled in backend

-   `$ua` is taken **directly from the HTTP User-Agent header**.

-   It's interpolated **without escaping or validation** inside a SQL query string.

-   If attacker crafts a User-Agent containing SQL syntax, it injects code into the query.

What happens:

-   The first WHERE clause looks for empty string (no results).

-   UNION injects a second query that pulls usernames and passwords from `user` table.

-   The trailing `'` is commented out by `#`, so no syntax error.

-   The results mix into the output attacker extracts sensitive data.
